# 51. TUI Visual Redesign

**Status:** Proposed
**Date:** 2025-01-26
**Deciders:** Product Team, Engineering Team
**Technical Story:** [TUI Visual Polish PRD](../product/features/tui-visual-polish.md) — compact header and input box design

---

## Context

The Forge TUI has three visual problems that make it feel heavy and unpolished compared to peer tools (Crush, Codex CLI, Gemini CLI):

1. **The header consumes 10 lines.** A 6-line block-character ASCII art logo followed by 4 lines of spacing and status bar means a full third of a standard 24-line terminal is occupied by chrome before a single message is visible. The hardcoded `headerHeight := 10` at `pkg/executor/tui/update.go:319` is load-bearing — it drives the viewport height calculation.

2. **The input box has a full 4-sided border.** The current `RoundedBorder()` on all sides creates a visual box around the input area. Competitors use open, borderless or rule-only designs that draw the eye to the content rather than the container.

3. **The bottom status bar is misaligned.** `buildBottomBar()` in `view.go` uses `len()` to measure styled strings when calculating padding. `len()` counts bytes including ANSI escape sequences, producing a wider measurement than the actual visible width. The result is a bottom bar that overflows or truncates unpredictably depending on the content and terminal width.

There is a fourth related issue: the tips/hints text is dense and static — the same four hints are shown regardless of whether the agent is idle, busy, or waiting for approval.

### Background

The ASCII art is generated by `pkg/ui/ascii.go` (`GenerateASCIIArt()`) using box-drawing characters (`█`, `╗`, `╚`, `═`). It is called from `pkg/executor/tui/view.go:buildHeader()`. The `headerHeight := 10` constant at `update.go:319` must match the actual rendered height of `buildHeader()` — any mismatch causes the viewport to be the wrong size.

Eleven input box alternatives were evaluated (documented in `docs/product/scratch/tui-input-alternatives.md`). Option B (top rule only) was selected after review.

Ten full-terminal layout mockups were produced (documented in `docs/product/scratch/tui-mockups.md`), all using Option B input.

### Problem Statement

The TUI's visual weight is disproportionate to its information density. New users see a large ASCII art block before any conversation content. The input box border adds visual noise. The bottom bar alignment breaks with certain model names or file paths. First impressions and day-to-day aesthetics are both below the quality bar set by direct competitors.

### Goals

- Replace the 6-line ASCII art header with a compact single-line header bar
- Replace the 4-sided bordered input box with a top-rule-only design (Option B)
- Fix the bottom bar padding calculation to use `lipgloss.Width()` instead of `len()`
- Replace the static tips line with a contextual hints line that adapts to TUI state

### Non-Goals

- Full TUI package decomposition (tracked separately in `docs/product/scratch/tui-v2.md`)
- Theme system / multiple colour schemes (future)
- Animated or dynamic header elements
- Changes to overlay visual styling (separate concern)

---

## Decision Drivers

* **Competitive positioning**: All three measured competitors use compact/absent headers and open input areas
* **Viewport space**: Recovering 7 lines of vertical space is significant on small terminals
* **Correctness**: The ANSI `len()` bug is a correctness issue, not just aesthetics
* **Containment**: All changes are in `view.go` and `styles.go` — no agent logic is touched

---

## Considered Options

### Option 1: Keep ASCII art, fix ANSI bug only

**Description:** Leave header and input box unchanged; fix only `buildBottomBar()` to use `lipgloss.Width()`.

**Pros:**
- Minimal change
- No risk of breaking header height calculation

**Cons:**
- Does not address the primary visual quality complaint
- New users still see a large logo block on first launch

### Option 2: Compact header + Option B input + ANSI fix (CHOSEN)

**Description:** Replace `buildHeader()` with a compact single-line bar. Replace the bordered input box style with a top-rule-only style. Fix `buildBottomBar()` padding. Replace the static tips with a state-aware hints function. Update `headerHeight` constant to match the new header height.

**Pros:**
- Recovers 7 lines of viewport space
- Visually consistent with market peers
- Fixes the ANSI correctness bug
- All changes contained in two files (`view.go`, `styles.go`)

**Cons:**
- `headerHeight` change is load-bearing — must be exact
- Existing ASCII art is removed from the visible path (though `pkg/ui/ascii.go` is retained)
- Users accustomed to the current look will see a different header

### Option 3: Full visual rewrite (new component system)

**Description:** Decompose the TUI into a component-per-zone architecture and rebuild the visual layer from scratch.

**Pros:**
- Cleanest long-term architecture
- Enables theme system

**Cons:**
- Weeks of work, high regression risk
- Scope far exceeds the current quality problem
- Belongs in `tui-v2.md` roadmap, not this PRD

---

## Decision

**Chosen Option:** Option 2 — Compact header + Option B input + ANSI fix

### Rationale

Option 2 delivers the full visual improvement with a contained, reviewable diff. The risk is managed by the fact that `headerHeight` is a single integer constant and the new header's line count is deterministic (always 2 lines: the bar and the separator). Option 3 is correct long-term but wrong now.

---

## Consequences

### Positive

- Conversation viewport gains 7+ lines on a 24-line terminal (from ~10 to ~17 usable lines)
- First impression is clean, minimal, professional
- Bottom bar aligns correctly at all terminal widths and with all model name lengths
- Hints line reduces cognitive load by showing only relevant keys for the current state

### Negative

- The ASCII art header is no longer visible in the default TUI path — users who liked it will notice
- `pkg/ui/ascii.go` becomes dead code in the TUI path (retained for potential use elsewhere)

### Neutral

- `headerHeight` changes from `10` to `3` — must be updated in one place (`update.go:319`)
- `buildHeader()` function is replaced in-place; callers do not change

## Implementation

### Step 1 — Compact header (view.go, ~25 lines)

Replace `buildHeader()` in `pkg/executor/tui/view.go`:

```go
// buildHeader renders the compact single-line header bar.
// Total height: 2 lines (bar + separator).
func (m model) buildHeader() string {
    // Truncate working directory to fit in available width
    cwd := m.workingDir
    if m.width > 0 && lipgloss.Width(cwd) > m.width/2 {
        cwd = "…" + cwd[len(cwd)-(m.width/2):]
    }

    left  := m.styles.brandStyle.Render("⬡ forge")
    mid   := m.styles.mutedStyle.Render(cwd)
    right := m.styles.mutedStyle.Render(m.modelName + "  " + version.Version)

    // Fill space between left, mid, right
    totalUsed := lipgloss.Width(left) + lipgloss.Width(mid) + lipgloss.Width(right)
    gap       := (m.width - totalUsed) / 2
    if gap < 1 { gap = 1 }
    pad       := strings.Repeat(" ", gap)

    bar       := left + pad + mid + pad + right
    separator := m.styles.dimStyle.Render(strings.Repeat("─", m.width))
    return bar + "\n" + separator
}
```

**`pkg/executor/tui/update.go:319`** — update `headerHeight`:

```go
// headerHeight is the number of lines occupied by buildHeader() output.
// Compact header: 1 line bar + 1 line separator = 2 lines.
// Plus 1 line for the contextual hints = 3 total.
headerHeight := 3
```

### Step 2 — Option B input box (styles.go + view.go, ~20 lines)

**`pkg/executor/tui/styles.go`** — replace `inputBoxStyle` with `inputRuleStyle`:

```go
// inputRuleStyle renders the horizontal rule above the input field (Option B).
inputRuleStyle = lipgloss.NewStyle().
    Foreground(lipgloss.Color(mutedGray)).
    Width(0) // width set dynamically in buildInputBox

// inputPromptStyle styles the ❯ glyph
inputPromptStyle = lipgloss.NewStyle().
    Foreground(lipgloss.Color(salmonPink)).
    Bold(true)
```

**`pkg/executor/tui/view.go`** — replace `buildInputBox()`:

```go
// buildInputBox renders the Option B input zone:
//   ──────────────────────── (rule, full width)
//   ❯ <textarea>
//   (blank line)
//   Enter · send  Alt+Enter · new line  / · commands  Ctrl+Y · copy  Ctrl+C
func (m model) buildInputBox() string {
    rule   := lipgloss.NewStyle().
                Foreground(lipgloss.Color(mutedGray)).
                Render(strings.Repeat("─", m.width))
    prompt := inputPromptStyle.Render("❯")
    input  := m.textarea.View()

    return rule + "\n" + prompt + " " + input
}
```

### Step 2b — Dynamic viewport height for multiline input (update.go, ~15 lines)

This is the most load-bearing change in the redesign. Under the existing bordered input box the
input zone has a fixed height. Under Option B the zone is:

```
rule              ← always 1 line
❯ <textarea>      ← grows with content (1 line minimum, N lines for multiline draft)
hints             ← always 1 line
```

`calculateViewportHeight()` must subtract the **actual textarea line count** from the available
height on every recalculation — not a fixed number.

**`pkg/executor/tui/update.go`** — updated `calculateViewportHeight()`:

```go
func (m *model) calculateViewportHeight() int {
    // headerHeight: bar + separator + hints line. Static — never grows.
    const headerHeight = 3

    // inputZoneHeight is dynamic: rule (1) + live textarea lines + hints (1).
    // Use live line count from the textarea value; cap at height/3 so a very
    // long draft cannot reduce the viewport to zero.
    liveLines := strings.Count(m.textarea.Value(), "\n") + 1
    if liveLines < 1 {
        liveLines = 1
    }
    maxInputLines := m.height / 3
    if maxInputLines < 1 {
        maxInputLines = 1
    }
    if liveLines > maxInputLines {
        liveLines = maxInputLines
    }
    inputZoneHeight := 1 + liveLines + 1  // rule + textarea + hints

    const statusBarHeight = 1

    available := m.height - headerHeight - inputZoneHeight - statusBarHeight
    if available < 1 {
        available = 1
    }
    return available
}
```

**Resize trigger**: call `m.recalculateLayout()` whenever the textarea content line count
changes. In the textarea update branch of `Update()`:

```go
prevLines := strings.Count(m.textarea.Value(), "\n")
newTextarea, cmd := m.textarea.Update(msg)
m.textarea = newTextarea
if strings.Count(m.textarea.Value(), "\n") != prevLines {
    m.recalculateLayout()
}
return m, cmd
```

This ensures the viewport shrinks when the user types a multiline draft and expands back when they
send the message and the textarea resets to one line.

### Step 3 — Fix bottom bar ANSI padding (view.go, ~5 lines)

**`pkg/executor/tui/view.go`** — in `buildBottomBar()`, replace every `len(styledStr)` with `lipgloss.Width(styledStr)`:

Current (broken):
```go
padding := m.width - len(leftSection) - len(rightSection)
```

Fixed:
```go
padding := m.width - lipgloss.Width(leftSection) - lipgloss.Width(rightSection)
if padding < 0 {
    padding = 0
}
```

Apply the same fix to every other `len()` call on a styled string in `buildBottomBar()` and `buildTokenDisplay()`.

### Step 4 — Contextual hints line (view.go, ~30 lines)

Replace the static `buildTips()` function with a state-aware version:

```go
// buildTips returns a single-line hints string adapted to the current TUI state.
func (m model) buildTips() string {
    muted := m.styles.mutedStyle

    switch {
    case m.overlay != nil:
        // An overlay is open — show overlay-specific hints
        return muted.Render("  Esc · close   Tab · next field   Enter · confirm")

    case m.agentRunning:
        // Agent is active
        hints := "  Ctrl+C · interrupt"
        if !m.followScroll {
            hints += "   G · follow output"
        }
        return muted.Render(hints)

    default:
        // Idle
        return muted.Render(
            "  Enter · send   Alt+Enter · new line   / · commands   Ctrl+Y · copy   Ctrl+C",
        )
    }
}
```

## Validation

### Success Metrics

- Manual on 24-line terminal: launch Forge → header occupies 2 lines, conversation viewport occupies ≥17 lines
- Manual: type in input field → top rule visible, no border box
- Manual: press `Alt+Enter` to add a second line to the draft → viewport height shrinks by 1 line to accommodate the taller input zone; no overflow or clipping
- Manual: send the message → textarea resets to 1 line, viewport height grows back; no layout jitter
- Manual: type a draft long enough to trigger soft-wrap → viewport adjusts dynamically, rule and hints remain visible
- Manual: resize terminal to narrow width → bottom bar does not overflow, right-aligned content truncates cleanly
- Manual: start an agent run → hints line shows interrupt hint, not send hint
- Manual: scroll up while agent runs → hints line shows "G · follow output"
- Regression: all existing keybindings continue to work
- Regression: `headerHeight` matches the actual pixel height of the new header (verify by counting newlines in `buildHeader()` output)

## Notes

**headerHeight is load-bearing but now static.** `headerHeight = 3` covers the compact header
bar, separator, and hints line — these never grow. The value at `update.go:319` must equal the
exact number of newline-separated lines returned by `buildHeader()` plus `buildTips()`. Count the
`\n` characters in the `assembleBaseView()` composition to verify before merging.

**inputZoneHeight is dynamic.** Unlike the old bordered input box (which had a fixed 3-line
height), Option B's input zone grows with the user's draft:

| User action | inputZoneHeight |
|---|---|
| Fresh / empty textarea | 3 (rule + 1 content line + hints) |
| One `Alt+Enter` pressed | 4 (rule + 2 content lines + hints) |
| N multiline lines | N + 2 |
| Very long draft (capped) | `min(liveLines, m.height/3) + 2` |

`calculateViewportHeight()` must recompute on every textarea height change. The textarea update
path in `Update()` must detect height changes and call `recalculateLayout()`. See Step 2b for the
full implementation. A unit test should verify that `calculateViewportHeight()` returns
`totalHeight - 3 - inputZoneHeight - 1` for a range of textarea line counts.

**lipgloss.Width vs len()**: `lipgloss.Width()` uses `go-runewidth` under the hood and correctly handles multi-byte UTF-8, wide CJK characters, and ANSI escape sequences. It should replace every `len()` call on any string that may contain styling or non-ASCII content.
